1 What was the problem? (the real pain)

These days data extraction is very common — OCR, LLMs and Document-AI systems extract information from PDFs, Word docs, scanned files and images.

But building a working extraction pipeline and scaling it efficiently to TB-scale are two completely different things.

⚠️ The real challenge When data reaches hundreds of GBs or TBs, the problem is no longer just "extraction quality". Core engineering problems become critical:
  • Parallelism
  • Orchestration
  • Memory utilization
  • Worker scaling
  • Infrastructure cost
  • Throughput
  • Distributed bottlenecks

They already had an extraction pipeline (for PDFs, DOCs, scanned docs, ZIPs, OCR-heavy files). The real problem was: scaling it efficiently for TB-scale.

As larger runs were estimated, the projected runtime and infra cost became so high that the existing architecture was economically hard to scale.

🎯 Core Objective:
1) Reduce infra cost   2) Improve extraction throughput   3) Make large-scale extraction operationally feasible.

2 Architecture of the Extraction System

The pipeline was built to process large enterprise document collections — PDFs, scanned docs, DOC/DOCX, ZIP archives, images, and OCR-heavy files.

High-level flow:

Source Systems / SharePoint / OneDrive / Graph API
▼
Distributed Extraction Instances (SageMaker, high-CPU)
▼
Worker-Level Processing (multiprocessing)
▼
OCR / Parsing / Extraction
▼
Output Storage (S3) + Metadata Tracking (Postgres)

The pipeline was deployed on distributed SageMaker instances using high-core CPU machines (e.g. ~192 vCPUs). Each instance processed a shard (subset of the dataset) independently — enabling horizontal scaling.

🧩 Inside each worker File Download (Graph/SharePoint) → Document Processing (PDF parse, OCR if needed, ZIP handling, image processing, DOC/DOCX parse) → Page-level OCR parallelism (~5 pages per thread) → Post-processing & Metadata.

3 The 3 Levels of Parallelism (where the game begins)

The most important aspect of the architecture — parallelism existed across multiple layers simultaneously.

1️⃣ Instance-Level Parallelism

The whole dataset was divided into shards and distributed across multiple SageMaker instances. This was the highest level of distributed execution. As load grew, more instances were added.

Dataset
   ├── Instance 1
   ├── Instance 2
   ├── Instance 3
   └── ...

2️⃣ Worker-Level Parallelism

Inside each instance, multiprocessing-based workers processed files in parallel (starting at ~30 workers/instance). Each worker handled: file download, parsing, OCR, extraction, metadata, output storage.

💡 Why multiprocessing? OCR workloads are CPU-bound. So multiprocessing was preferred over threading — to properly utilize CPU cores (bypassing the GIL).

3️⃣ Parallelism Inside Each Worker

The worker itself had yet another parallelism layer. For large PDFs/scanned docs: pages processed independently, OCR run concurrently across pages (~5 OCR page threads), and image preprocessing added extra threading.

⚠️ Effective concurrency The runtime model roughly became: 
Distributed Instances
    × 150 Workers
    × Multiple Files
    × ~5 OCR Page Threads
Meaning the actual runtime concurrency was far larger than the configured worker count!

At small scale: throughput was great. At large scale: CPU oversubscription, thread contention, memory pressure, scheduling overhead, long-tail behaviour, and orchestration bottlenecks.

4 Profiling — Understanding the System

Before optimizing, it was essential to understand how the system actually used compute resources during large runs.

🤔 Core Question: Are the instances using available CPU + memory efficiently, or are hidden bottlenecks inflating runtime and cost?

Profiling was added at 3 levels: instance level, worker level, and file-processing level. Since deep cloud-native AWS observability was limited, most profiling was done in the application layer via Python instrumentation.

🛠️ Tools
  • psutil
  • Custom worker instrumentation
  • Profiling tables
  • Run/file-level tracing → Postgres
📏 Metrics tracked
  • CPU utilization (start/mid/end)
  • Memory usage + RSS memory (MB)
  • Worker execution time
  • Long-running files (long-tail)
  • Shard-level & retry/error behaviour

Common psutil functions:

psutil.Process().memory_info().rss
psutil.Process().cpu_percent()
psutil.virtual_memory()
psutil.cpu_percent()
💡 Early finding CPU utilization was surprisingly LOW — despite high runtime and infra usage! This meant compute was sitting idle while something else was blocking.

Profiling revealed: underutilized CPU, long-tail files, uneven worker behaviour, nested concurrency effects, and orchestration bottlenecks — none of which were visible from overall runtime alone.

5 Scaling Iterations & Results

After profiling, the next step was straightforward: increase parallelism, improve throughput.

~30 workers/instance  →  CPU peak only ~20% 😭
▼ aggressive scale
~150 workers/instance  →  CPU peak ~50% only

At first runtime improved noticeably. But as workloads grew, behaviour became inconsistent — some runs improved, some plateaued, some got slower.

⚠️ Expectation vs Reality Expectation: more instances + more workers = lower runtime
Reality: distributed runs became unpredictable. CPU still only ~50% — so the bottleneck was deeper than raw compute.
💡 Inverse pattern: Worker Starvation Larger workloads on the same instance were completing more efficiently than smaller ones! As load increased:
  • CPU utilization got healthier
  • Worker utilization improved
  • Throughput became more stable
Meaning the system behaved better under sustained load than bursty/partial load. Some workers stayed busy for very long, a few files took ~1 hour!

➜ This no longer looked like a compute bottleneck.

6 The Single-Instance Contradiction (turning point!)

The most important experiment: running a ~40 GB workload on a single high-CPU instance.

⚡ The entire workload completed in just ~70 minutes!
And files that looked like "problem files" in distributed runs completed smoothly here, without delays.

This changed the investigation's direction entirely. The bottleneck was coming from orchestration and concurrency behaviour, not OCR speed.

🔄 Investigation shift Old question: "How do we increase parallelism?"
New question: "What hidden bottlenecks appear under distributed scale?"

7 Assumptions, Experiments & Dead-Ends

As scaling became inconsistent, the journey turned highly experimental. Many assumptions were tested before the real bottleneck became clear.

🔍 Hypothesis #1 — Is OCR the Bottleneck?

First suspicion: OCR itself was slowing things down. Multiple engines tested: Tesseract/pytesseract, PaddleOCR, Surya, EasyOCR, DocTR.

  • Surprisingly, pytesseract performed best — especially for CPU-intensive extraction.
  • And files that looked "stuck" in distributed runs completed in minutes in isolated OCR tests.
💡 Result OCR was not the primary bottleneck. But OCR threading / process contention / nested OCR parallelism were still suspects.

⚙️ Hypothesis #2 — OCR Threading & Nested Parallelism

OCR thread tuning (OMP_THREAD_WORKER=1), concurrency adjustments, reducing nested OCR pressure.

⚠️ Result Little to no meaningful improvement. Runtime stayed inconsistent, some runs even regressed.

🧵 Hypothesis #3 — Multiprocessing / Orchestration Blocking

Changed execution handling: pool.map() → submit() + as_completed() — for finer control, better handling of long-running tasks, and orchestration visibility.

⚠️ Result Throughput didn't meaningfully improve, sometimes slight regression. So task scheduling alone wasn't the root bottleneck.

🐍 Hypothesis #4 — Python Multiprocessing & GIL Limits

Focus shifted to Python: multiprocessing overhead, GIL behaviour, inter-process coordination, large-scale multiprocessing limits.

📌 Note Python limitations contributed, but were not the primary reason. The issue persisted even with process-based parallelism.

⚡ PySpark Experiments

The workload now resembled a distributed data-processing problem, so PySpark was tried — assuming Spark's orchestration would improve scaling.

⚠️ Result Results were almost identical to the existing architecture. The expected breakthrough never came — so deeper PySpark migration was not pursued.
🚧 More experiments, more contradictions OCR tracing, reducing nested concurrency, identifying blocking behaviour, isolating long-tail — none consistently solved the scaling degradation.
💡 Big realisation: The bottleneck came not from an isolated component, but from the interactions between multiple system layers.

📋 All major assumptions were tested extensively — but no single change consistently solved the problem. The problem was multi-layered and complex.

8 Hidden Bottlenecks (the real villains!)

The biggest contradiction came from scaling behaviour itself: 30→150 workers boosted throughput, single-instance ~40GB finished in ~70 min — yet the same architecture degraded at distributed scale.

The issue was no longer: OCR throughput, CPU saturation, memory exhaustion, or inefficient multiprocessing alone. Instead the system showed: hidden contention, chained delays, distributed bottlenecks — that only appeared at scale.

🗄️ 1) DB Logging Contention

Database activity inside active extraction workers was a major bottleneck. Workers were continuously: writing logs, updating progress rows, interacting with Postgres — during execution.

⚠️ At scale Across ~150 workers × multiple instances, DB ops created: contention, blocking, repeated connection overhead, execution delays. Postgres became part of the critical execution path — instead of just an observability layer!

🔄 2) Repeated Client & Session Setup

Instead of doing actual extraction work, workers repeatedly paid for: Graph client setup, S3 client creation, session initialization, and per-file setup operations.

⚠️ Individually small. But thousands of files × hundreds of workers × distributed instances = significant waste.

🌐 3) Extra Graph / Metadata Calls

The pipeline had avoidable Graph-side overhead — some paths made per-file extra metadata calls, validation calls, or recovery requests.

⚠️ One extra call/file sounds harmless. But at large scale these became thousands of extra requests, higher upstream pressure, and contention.

🚦 4) Graph Concurrency & Retry Amplification

The biggest scale-out issue: worker count translated directly into Graph download concurrency.

Many instances × many workers = aggressive request bursts to Graph/SharePoint → 429 throttling, retries, worker blocking, inflated wall-clock runtime.

⚠️ Retry amplification A throttled request doesn't fail immediately — the worker stays busy waiting + retrying, creating another wave of upstream pressure. 😭
💡 Major realisation:
Extraction parallelism and download parallelism cannot scale identically!

9 Why did the Bottlenecks become Expensive?

The real problem: these costs were nested and compounding. Inside one worker, a single file involved:

Graph download → OCR → retries → DB logging → client setup → output handling
▼ repeat
× ~150 workers
▼ repeat
× multiple distributed instances
🧮 Compounding effect There was no single catastrophic bottleneck. Several small inefficiencies were interacting together:
  • Graph throttling
  • Retry amplification
  • DB contention
  • Orchestration overhead
  • Nested concurrency pressure
That's what made the pipeline so hard to diagnose.

Small cost/file × 150 workers × N instances × upstream pressure (429s) × blocked orchestration ＝ 🔥 RUNTIME EXPLODES (hours instead of minutes)

10 Breakthrough Architecture (The Fix)

After identifying the hidden bottlenecks, the architecture was redesigned around one core principle:

Keep extraction highly parallel, but CONTROL the expensive operations around it.

🚦 Bounded Graph Concurrency

Problem: increasing worker count unintentionally increased Graph request fanout too → 429 throttling, retry amplification, blocked workers.

Solution: semaphore-based request control to independently cap Graph download concurrency — while keeping workers (~150) highly parallel.

multiprocessing.Manager().BoundedSemaphore()
💡 Idea Instead of every worker independently triggering downloads, a Graph request first acquires a semaphore slot, then hits the upstream service. So download concurrency doesn't explode with worker scale.

Optimized execution flow:

Bounded Graph Download Queue
▼
Reusable Worker-local Clients
▼
Persistent Session-based Downloads
▼
OCR + Extraction
▼
Reduced DB Logging
▼
Output Upload + Completion

🔄 Optimizing the Request Layer

The Graph request layer was made resilient for distributed execution:

  • Reusable worker-local Graph/S3 clients
  • Persistent requests.Session() usage
  • Timeout-aware downloads
  • Retry-aware execution
  • Token refresh handling
  • Jittered exponential backoff

Instead of recreating an HTTP session per file, workers reused persistent session state locally → less connection setup, less repeated auth, less TCP/HTTP churn, less latency.

⚠️ Retry Amplification & Jitter Handling

The optimized retry layer handled: 429, transient 5xx, request timeouts, token expiry. The logic respected: Retry-After headers, exponential windows, and randomized jitter.

💡 Why jitter? Without jitter, after a throttling event the entire distributed fleet retries simultaneously → another burst! Randomized jitter significantly stabilized request behaviour across instances.

🗄️ Reducing Worker Overhead

Expensive ops inside active extraction workers were reduced. The optimized workers:

  • Reused Graph & S3 clients locally
  • Reduced repeated setup work
  • Minimized unnecessary metadata calls
  • Removed heavy DB logging from the active execution path
✅ Benefit Instead of frequent Postgres updates during extraction, DB interaction was greatly reduced → preventing: shared-row contention, DB session/run blocking, and high-concurrency worker delays.

13 Engineering Lessons — Top 10 (remember these!)

1️⃣ More workers help — if orchestrated well. Throughput improves only when workers coordinate efficiently, with minimal overhead & contention.
2️⃣ Distributed bottlenecks often appear only at scale. Systems that look fine in small tests can degrade badly as data volume + concurrency grow.
3️⃣ Balance parallelism & threading with care. Nested concurrency (workers × OCR threads) must be balanced — else contention, resource exhaustion, diminishing returns.
4️⃣ Upstream services are bounded, shared resources. Respect limits, quotas, capacity. Efficient usage + backoff strategies prevent throttling/failures.
5️⃣ Retry systems without jitter create "retry storms". Uncoordinated retries amplify load and delay recovery. Use: jitter, exponential backoff, circuit breakers.
6️⃣ Small per-unit overhead becomes massive at scale. Milliseconds per operation → hours/days of wasted compute when repeated thousands of times.
7️⃣ Machine saturation & workload density heavily affect throughput efficiency. Sustained execution > bursty distributed scaling (better utilization, less orchestration overhead).
8️⃣ Profile EVERYTHING, not just CPU. Usage, waits, latency, I/O, network, locks, garbage collection, external dependencies — identify end-to-end.
9️⃣ Unoptimized DB/IO create exponential impact. Chatty, unindexed, or frequent operations on the same shared resource → bottlenecks, blocks, cascading delays.
🔟 Stable scaling is a system-wide discipline. Coordinated control over orchestration, retries, concurrency, resource usage & execution paths = predictable performance at any scale.
🧠 The Biggest Realisation:
Large-scale distributed pipelines are usually limited not by raw compute, but by orchestration efficiency and contention amplification.

14 Wrapping Up

This was genuinely a difficult engineering problem. ⚠️

  • ⭐ Many assumptions failed.
  • ⭐ Several experiments regressed performance instead of improving it.
  • ⭐ Some bottlenecks only appeared after fixing earlier ones.
  • ⭐ And many findings initially looked contradictory.

But that's exactly what made the journey valuable. Beyond runtime + cost improvements, the most valuable outcome was: a deeper understanding of how large-scale distributed systems actually behave under pressure.

✅ Who are these learnings for?
  • People building OCR pipelines
  • Distributed processing workloads
  • Large-scale AI extraction platforms
So they can avoid the same pitfalls, and think about scale from a systems perspective, not just a compute perspective.
"Scale isn't a compute problem — it's a systems-design discipline."
