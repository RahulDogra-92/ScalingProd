1 Problem kya tha? (the real pain)

Aaj kal data extraction bohot common ho gaya hai — OCR, LLMs aur Document-AI systems PDFs, Word docs, scanned files aur images se information nikaalte hain.

Lekin ek working extraction pipeline banana aur usko TB-scale tak efficiently scale karna — ye do bilkul alag cheezein hain.

⚠️ Asli challenge Jab data hundreds of GBs ya TBs me pahunchta hai, tab problem sirf "extraction quality" nahi rehti. Core engineering problems important ho jaati hain:
  • Parallelism
  • Orchestration
  • Memory utilization
  • Worker scaling
  • Infrastructure cost
  • Throughput
  • Distributed bottlenecks

Unke paas pehle se ek extraction pipeline thi (PDFs, DOCs, scanned docs, ZIPs, OCR-heavy files ke liye). Asli problem thi: isko TB-scale ke liye efficiently scale karna.

Jaise-jaise bade runs estimate kiye gaye, projected runtime aur infra cost itni high ho gayi ki existing architecture economically scale karna mushkil tha.

🎯 Core Objective:
1) Infra cost kam karo   2) Extraction throughput badhao   3) Large-scale extraction ko operationally feasible banao.

2 Extraction System ki Architecture

Pipeline bani thi bade enterprise document collections process karne ke liye — PDFs, scanned docs, DOC/DOCX, ZIP archives, images, aur OCR-heavy files.

High-level flow:

Source Systems / SharePoint / OneDrive / Graph API
▼
Distributed Extraction Instances (SageMaker, high-CPU)
▼
Worker-Level Processing (multiprocessing)
▼
OCR / Parsing / Extraction
▼
Output Storage (S3) + Metadata Tracking (Postgres)

Pipeline distributed SageMaker instances par deploy thi, high-core CPU machines (e.g. ~192 vCPUs) use hoti thi. Har instance ek shard (dataset ka subset) independently process karta tha — isse horizontal scaling possible hua.

🧩 Inside each worker File Download (Graph/SharePoint) → Document Processing (PDF parse, OCR if needed, ZIP handling, image processing, DOC/DOCX parse) → Page-level OCR parallelism (~5 pages per thread) → Post-processing & Metadata.

3 Parallelism ke 3 Levels (yahin se khel shuru)

Architecture ki sabse important baat — parallelism ek saath multiple layers par exist karta tha.

1️⃣ Instance-Level Parallelism

Pura dataset shards me divide hota tha, multiple SageMaker instances par distribute. Ye sabse upar ka distributed execution level tha. Load badhta to instances add karte.

Dataset
   ├── Instance 1
   ├── Instance 2
   ├── Instance 3
   └── ...

2️⃣ Worker-Level Parallelism

Har instance ke andar multiprocessing-based workers files parallel process karte the (~30 workers/instance se shuru). Har worker khud handle karta: file download, parsing, OCR, extraction, metadata, output storage.

💡 Kyun multiprocessing? OCR workload CPU-bound hota hai. Isliye threading ke bajaye multiprocessing choose kiya — taaki CPU cores theek se use ho (GIL bypass ho).

3️⃣ Parallelism Inside Each Worker

Worker ke andar bhi ek aur parallelism layer thi. Bade PDFs/scanned docs ke liye: pages independently process, OCR concurrently across pages (~5 OCR page threads), aur image preprocessing me extra threading.

⚠️ Effective concurrency Runtime model roughly ban gaya: 
Distributed Instances
    × 150 Workers
    × Multiple Files
    × ~5 OCR Page Threads
Matlab actual runtime concurrency configured worker count se kahin zyada thi!

Small scale par: throughput bohot accha. Bade scale par: CPU oversubscription, thread contention, memory pressure, scheduling overhead, long-tail behaviour, aur orchestration bottlenecks.

4 Profiling — System ko samajhna

Optimize karne se pehle ye samajhna zaroori tha ki system bade runs me compute resources kaise use kar raha hai.

🤔 Core Question: Kya instances available CPU + memory efficiently use kar rahe hain, ya hidden bottlenecks runtime aur cost ko inflate kar rahe hain?

Profiling 3 levels par add hui: instance level, worker level, aur file-processing level. Kyunki deep cloud-native AWS observability limited thi, isliye zyadatar profiling application layer me Python instrumentation se ki gayi.

🛠️ Tools
  • psutil
  • Custom worker instrumentation
  • Profiling tables
  • Run/file-level tracing → Postgres
📏 Metrics tracked
  • CPU utilization (start/mid/end)
  • Memory usage + RSS memory (MB)
  • Worker execution time
  • Long-running files (long-tail)
  • Shard-level & retry/error behaviour

Common psutil functions:

psutil.Process().memory_info().rss
psutil.Process().cpu_percent()
psutil.virtual_memory()
psutil.cpu_percent()
💡 Early finding CPU utilization surprisingly LOW tha — bavjood iske ki runtime aur infra usage high tha! Iska matlab compute idle pada tha, kuch aur block kar raha tha.

Profiling se mila: underutilized CPU, long-tail files, uneven worker behaviour, nested concurrency effects, aur orchestration bottlenecks — jo sirf overall runtime dekh ke visible nahi the.

5 Scaling Iterations & Results

Profiling ke baad agla step seedha tha: parallelism badhao, throughput improve karo.

~30 workers/instance  →  CPU peak sirf ~20% 😭
▼ aggressive scale
~150 workers/instance  →  CPU peak ~50% only

Pehle runtime noticeably improve hua. Lekin jaise workload badha, behaviour inconsistent ho gaya — kuch runs improve, kuch plateau, kuch slower.

⚠️ Expectation vs Reality Expectation: more instances + more workers = lower runtime
Reality: distributed runs unpredictable ho gaye. CPU abhi bhi sirf ~50% — toh bottleneck raw compute se deeper tha.
💡 Ulta pattern: Worker Starvation Bade workloads same instance par chote workloads se zyada efficiently complete ho rahe the! Load badhne par:
  • CPU utilization healthier
  • Worker utilization better
  • Throughput more stable
Matlab system sustained load me behtar behave karta tha, bursty/partial load me nahi. Kuch workers bohot der tak busy, kuch files ~1 ghanta tak le rahi thi!

➜ Ye compute bottleneck jaisa lag hi nahi raha tha.

6 The Single-Instance Contradiction (turning point!)

Sabse important experiment: ~40 GB workload ko ek single high-CPU instance par chalaya.

⚡ Pura workload sirf ~70 minutes me complete!
Aur jo files distributed runs me "problem files" lag rahi thi — wo yahan smoothly bina delay complete ho gayi.

Isne investigation ki direction hi badal di. Bottleneck OCR speed se nahi, balki orchestration aur concurrency behaviour se aa raha tha.

🔄 Investigation shift Purana sawaal: "Parallelism kaise badhayein?"
Naya sawaal: "Distributed scale par kaunse hidden bottlenecks ban rahe hain?"

7 Assumptions, Experiments & Dead-Ends

Scaling inconsistent hone par journey bohot experimental ho gayi. Asli bottleneck samajhne se pehle kaafi assumptions test hui.

🔍 Hypothesis #1 — OCR hi Bottleneck hai?

Pehla shak: OCR khud slow kar raha hai. Multiple engines test kiye: Tesseract/pytesseract, PaddleOCR, Surya, EasyOCR, DocTR.

  • Surprisingly pytesseract hi best perform kiya — khaaskar CPU-intensive extraction me.
  • Aur jo files distributed runs me "stuck" lag rahi thi, wo isolated OCR tests me minutes me complete ho gayi.
💡 Result OCR primary bottleneck nahi tha. Par OCR threading / process contention / nested OCR parallelism abhi bhi suspect the.

⚙️ Hypothesis #2 — OCR Threading & Nested Parallelism

OCR thread tuning (OMP_THREAD_WORKER=1), concurrency adjustments, nested OCR pressure kam karna.

⚠️ Result Little to no meaningful improvement. Runtime inconsistent raha, kuch runs to regress bhi hue.

🧵 Hypothesis #3 — Multiprocessing / Orchestration Blocking

Execution handling badli: pool.map() → submit() + as_completed() — taaki finer control, long-running tasks ka better handling, aur orchestration visibility mile.

⚠️ Result Throughput meaningfully improve nahi hua, kabhi-kabhi slight regression. Matlab task scheduling alone root bottleneck nahi tha.

🐍 Hypothesis #4 — Python Multiprocessing & GIL Limits

Focus Python par shift: multiprocessing overhead, GIL behaviour, inter-process coordination, large-scale multiprocessing limits.

📌 Note Python limitations contribute karte the, par primary reason nahi. Issue process-based parallelism ke saath bhi bana raha.

⚡ PySpark Experiments

Workload ab distributed data-processing problem jaisa lag raha tha, toh PySpark try kiya — sochke ki Spark ki orchestration scaling improve karegi.

⚠️ Result Results existing architecture jaise hi the. Expected breakthrough nahi aaya — isliye deeper PySpark migration pursue nahi kiya.
🚧 Aur experiments, aur contradictions OCR tracing, nested concurrency reduce karna, blocking behaviour identify karna, long-tail isolate karna — kisi ne bhi consistently scaling degradation solve nahi ki.
💡 Big realisation: Bottleneck ek isolated component se nahi, balki multiple system layers ke beech ke interactions se aa raha tha.

📋 Saari major assumptions extensively test hui — par koi bhi single change consistently problem solve nahi kar paaya. Problem multi-layered aur complex thi.

8 Hidden Bottlenecks (asli villain log!)

Sabse bada contradiction scaling behaviour se hi aaya: 30→150 workers ne throughput badhaya, single-instance ~40GB ~70 min me hua — par wahi architecture distributed scale par degrade kar raha tha.

Issue ab ye nahi tha: OCR throughput, CPU saturation, memory exhaustion, ya inefficient multiprocessing alone. Balki system me dikhe: hidden contention, chained delays, distributed bottlenecks — jo sirf scale par aate the.

🗄️ 1) DB Logging Contention

Active extraction workers ke andar database activity ek bada bottleneck thi. Workers continuously: logs likhte, progress rows update karte, Postgres ke saath interact karte — execution ke during.

⚠️ Scale par ~150 workers × multiple instances par DB ops ne create kiya: contention, blocking, repeated connection overhead, execution delays. Postgres critical execution path ka hissa ban gaya — observability layer ke bajaye!

🔄 2) Repeated Client & Session Setup

Actual extraction work ke bajaye workers baar-baar cost de rahe the: Graph client setup, S3 client creation, session initialization, aur per-file setup operations.

⚠️ Individually chhota lagta hai. Par hazaaron files × hundreds of workers × distributed instances = significant waste.

🌐 3) Extra Graph / Metadata Calls

Pipeline me avoidable Graph-side overhead tha — kuch paths per-file extra metadata calls, validation calls, ya recovery requests karte the.

⚠️ Ek extra call/file harmless lagta hai. Par large-scale par ye hazaaron extra requests, higher upstream pressure, aur contention ban gaye.

🚦 4) Graph Concurrency & Retry Amplification

Sabse bada scale-out issue: worker count directly Graph download concurrency me translate ho raha tha.

Bahut instances × bahut workers = Graph/SharePoint par aggressive request bursts → 429 throttling, retries, worker blocking, inflated wall-clock runtime.

⚠️ Retry amplification Throttled request turant fail nahi hoti — worker busy rehta hai wait + retry karte hue, jo ek aur upstream pressure wave create karta hai. 😭
💡 Major realisation:
Extraction parallelism aur download parallelism identically scale nahi kar sakte!

9 Ye Bottlenecks Expensive kyun bane?

Asli problem: ye costs nested aur compounding thi. Ek worker me, ek single file involve karti thi:

Graph download → OCR → retries → DB logging → client setup → output handling
▼ repeat
× ~150 workers
▼ repeat
× multiple distributed instances
🧮 Compounding effect Koi ek catastrophic bottleneck nahi tha. Kai chhoti inefficiencies ek saath interact kar rahi thi:
  • Graph throttling
  • Retry amplification
  • DB contention
  • Orchestration overhead
  • Nested concurrency pressure
Isi liye ye pipeline diagnose karna itni mushkil thi.

Small cost/file × 150 workers × N instances × upstream pressure (429s) × blocked orchestration ＝ 🔥 RUNTIME EXPLODES (hours instead of minutes)

10 Breakthrough Architecture (The Fix)

Hidden bottlenecks identify karne ke baad architecture ek core principle par redesign hua:

Extraction ko highly parallel rakho, par uske around ki expensive operations ko CONTROL karo.

🚦 Bounded Graph Concurrency

Problem: worker count badhane se Graph request fanout bhi unintentionally badh jaata tha → 429 throttling, retry amplification, blocked workers.

Solution: semaphore-based request control se Graph download concurrency ko independently cap kiya — workers (~150) ko highly parallel rakhte hue.

multiprocessing.Manager().BoundedSemaphore()
💡 Idea Har worker independently download trigger karne ke bajaye, Graph request pehle ek semaphore slot acquire karti hai, tabhi upstream service hit karti hai. Isse download concurrency worker-scale ke saath explode nahi hoti.

Optimized execution flow:

Bounded Graph Download Queue
▼
Reusable Worker-local Clients
▼
Persistent Session-based Downloads
▼
OCR + Extraction
▼
Reduced DB Logging
▼
Output Upload + Completion

🔄 Optimizing the Request Layer

Graph request layer ko distributed execution me resilient banaya:

  • Reusable worker-local Graph/S3 clients
  • Persistent requests.Session() usage
  • Timeout-aware downloads
  • Retry-aware execution
  • Token refresh handling
  • Jittered exponential backoff

Har file par HTTP session recreate karne ke bajaye, workers persistent session state locally reuse karte the → kam connection setup, kam repeated auth, kam TCP/HTTP churn, kam latency.

⚠️ Retry Amplification & Jitter Handling

Optimized retry layer handle karta tha: 429, transient 5xx, request timeouts, token expiry. Logic respect karta tha: Retry-After headers, exponential windows, aur randomized jitter.

💡 Jitter kyun? Bina jitter ke, throttling event ke baad pura distributed fleet ek saath retry karta hai → ek aur burst! Randomized jitter ne request behaviour ko across instances kaafi stabilize kiya.

🗄️ Reducing Worker Overhead

Active extraction workers ke andar expensive ops kam kiye. Optimized workers:

  • Graph & S3 clients locally reuse karte the
  • Repeated setup work reduce
  • Unnecessary metadata calls minimize
  • Heavy DB logging ko active execution path se hata diya
✅ Faayda Extraction ke during frequent Postgres updates ke bajaye DB interaction kaafi kam kiya → prevent: shared-row contention, DB session/run blocking, aur high-concurrency worker delays.

13 Engineering Lessons — Top 10 (yaad rakhna!)

1️⃣ More workers help — agar orchestration achhi ho. Throughput tabhi improve hota hai jab workers efficiently coordinate hon, minimal overhead & contention ke saath.
2️⃣ Distributed bottlenecks aksar sirf scale par dikhte hain. Jo systems chote tests me theek lagte hain, wo data volume + concurrency badhne par badly degrade kar sakte hain.
3️⃣ Parallelism & threading ko carefully balance karo. Nested concurrency (workers × OCR threads) ko balance karna zaroori — warna contention, resource exhaustion, diminishing returns.
4️⃣ Upstream services bounded, shared resources hain. Limits, quotas, capacity ko respect karo. Efficient usage + backoff strategies throttling/failures prevent karte hain.
5️⃣ Retry systems bina jitter ke "retry storms" banate hain. Uncoordinated retries load amplify karke recovery delay karti hain. Use: jitter, exponential backoff, circuit breakers.
6️⃣ Small per-unit overhead scale par massive ban jaata hai. Per operation milliseconds → hours/days of wasted compute, jab hazaaron baar repeat ho.
7️⃣ Machine saturation & workload density throughput efficiency ko heavily affect karte hain. Sustained execution > bursty distributed scaling (better utilization, kam orchestration overhead).
8️⃣ Profile EVERYTHING, sirf CPU nahi. Usage, waits, latency, I/O, network, locks, garbage collection, external dependencies — end-to-end identify karo.
9️⃣ Unoptimized DB/IO exponential impact create karte hain. Chatty, unindexed, ya frequent operations same shared resource par → bottlenecks, blocks, cascading delays.
🔟 Stable scaling ek system-wide discipline hai. Orchestration, retries, concurrency, resource usage & execution paths par coordinated control = predictable performance at any scale.
🧠 The Biggest Realisation:
Large-scale distributed pipelines aksar raw compute se nahi, balki orchestration efficiency aur contention amplification se limited hote hain.
